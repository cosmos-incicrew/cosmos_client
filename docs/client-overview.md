cosmos_client — 앱 구성 설계 (팀 공유)

작성 금별, 2026년 7월 10일. cosmos 팀 공유용 문서입니다. 프론트 클라이언트 앱을 어떤 구조로 짰는지 팀원이 이해하고, 각자 화면을 이 구조 위에서 붙일 수 있도록 정리했습니다. 화면 디자인은 피그마, 데이터와 API 계약은 백엔드 문서와 API 명세서가 기준입니다.


개요

이 앱은 화면과 사용자 입력, 로그인(Supabase Auth), 서버 호출만 담당합니다. 비즈니스 로직과 데이터 조회, LLM 생성은 전부 백엔드(cosmos_server)가 맡습니다. 클라이언트는 서버가 어디 있는지 주소만 알고 호출하며, 서버 코드를 품지 않습니다.

구조는 Flutter의 표준 방식인 feature-first를 그대로 따르되, 기능 폴더를 피그마의 화면 흐름에 맞춰 구성했습니다. 백엔드 API가 아직 대부분 미구현(501 스텁) 상태이기 때문에, 화면은 목업 데이터로 먼저 완성하고 API가 확정되면 데이터 계층만 교체하는 방식으로 갑니다. 이렇게 하면 백엔드를 기다리지 않고도 프론트가 화면을 만들어 둘 수 있습니다.


기술 스택

프레임워크는 Flutter(Dart 3.6 이상)를 씁니다. 단일 코드로 iOS, 안드로이드, 웹을 모두 커버할 수 있어서 선택했습니다. 상태 관리는 Riverpod을 쓰는데, 의존성 주입과 테스트, 목업 교체가 쉬워서입니다. 화면 이동(라우팅)은 go_router로 하며, 인증 상태에 따른 화면 전환을 한 곳에서 관리합니다. 네트워킹은 두 가지를 씁니다. 로그인과 세션은 supabase_flutter가, 백엔드 REST 호출은 dio가 담당합니다. 로컬 저장소는 Hive(민감하지 않은 캐시용)와 flutter_secure_storage(토큰 같은 민감 데이터용)를 나눠서 씁니다. 디자인은 Material 3에 커스텀 테마를 얹었습니다.


폴더 구조

코드는 세 덩어리로 나뉩니다.

첫째, app 폴더는 앱을 조립하는 곳입니다. 여기에는 앱 루트 위젯, 라우터(화면 이동과 인증 리다이렉트, 하단 탭), 테마(색상과 Material 3 스타일)가 들어갑니다. 실제 기능 작업에서는 자주 건드릴 일이 없는 배선 계층입니다.

둘째, core 폴더는 여러 기능이 함께 쓰는 공용 인프라입니다. 환경 설정(Env), Supabase와 Dio 클라이언트, 로컬 저장소, 그리고 로딩·에러·빈 상태 같은 공통 위젯이 들어갑니다.

셋째, features 폴더가 실제 작업 공간입니다. 기능별로 폴더가 나뉘며, 각 폴더는 다시 data(모델과 저장소)와 presentation(화면, 위젯, 상태 관리자)으로 나뉩니다. 현재 기능 폴더는 다음과 같습니다. auth는 로그인(카카오·네이버·구글·게스트), onboarding은 서비스 소개에서 프로필, 피부고민 등록까지, home은 기능 허브, bsti는 16타입 검사와 결과, my_shelf는 나의 화장대(제품·성분 검색·등록·점수), recommendation은 맞춤 추천입니다. 여기에 성분과 제품 데이터 모델을 담는 ingredient와 product 폴더, 마이페이지인 profile이 있습니다.


설계 원칙 세 가지

이 세 가지는 PR 리뷰에서 가장 먼저 확인하는 항목입니다.

첫째, 의존 방향은 presentation에서 data로 한 방향입니다. 화면에서 서버나 저장소를 직접 호출하지 않고 반드시 저장소(repository)를 거칩니다. 이렇게 해야 목업을 실제 API로 바꿀 때 화면 코드가 바뀌지 않습니다. 백엔드의 router에서 service, schemas로 흐르는 단방향과 같은 사상입니다.

둘째, 기능끼리 서로 직접 import하지 않습니다. 공유가 필요하면 그 코드를 core로 올린 뒤 양쪽이 참조합니다. 다섯 명이 동시에 작업할 때 기능이 서로 얽히면 충돌과 회귀가 생기기 때문입니다. 백엔드는 이 규칙을 import-linter 도구로 강제하지만, 프론트는 자동 도구가 없으므로 리뷰로 강제합니다.

셋째, 외부 접근(Supabase, 서버)은 반드시 core의 네트워크 클라이언트를 거칩니다. 화면이나 저장소에서 Dio나 Supabase 인스턴스를 새로 만들지 않습니다. 그래야 인증과 에러 처리가 한 방식으로 통일됩니다.


화면 흐름

사용자는 먼저 로그인 화면을 만납니다. 카카오·네이버·구글로 시작하거나, 로그인 없이 게스트로 둘러볼 수 있습니다. 로그인 후에는 온보딩으로 이어집니다. 서비스 소개를 보고, 프로필(닉네임·나이·성별)을 입력하고, 피부고민을 선택합니다. 그다음 홈에 도착합니다.

홈은 기능들의 허브입니다. 여기서 세 방향으로 갈라집니다. BSTI 검사로 가면 검사를 시작해 25문항에 답하고, 분석을 거쳐 결과(피부타입 코드, 네 축의 비율, 권장 성분과 주의 성분)를 봅니다. 나의 화장대로 가면 등록한 제품·성분 목록과 화장대 점수를 보고, 검색으로 제품이나 성분을 추가합니다. 맞춤 추천으로 가면 프로필과 BSTI 결과, 화장대를 바탕으로 성분과 제품을 추천받습니다. 하단 탭은 홈, 화장대, 마이 세 개입니다.

현재 상태는 이렇습니다. 위 화면 폴더가 모두 만들어져 있고 화면 사이의 이동(라우팅)이 실제로 동작합니다. 다만 각 화면 속은 아직 "준비 중" 플레이스홀더이며(핵심인 BSTI 결과 화면만 목업 데이터로 실제 표시), 상세한 화면 UI는 담당자가 피그마대로 채웁니다.


기능과 서버 모듈의 대응

프론트 기능은 백엔드 모듈과 다음과 같이 짝지어집니다. auth는 Supabase Auth가 직접 처리하므로 서버 모듈이 없습니다. onboarding은 사용자 프로필 관련 API(/api/v1/users)와 연결됩니다. bsti는 서버 bsti 모듈(/api/v1/bsti), my_shelf는 ingredient_search와 ingredient_detail 모듈(/api/v1/ingredients), recommendation은 recommendation 모듈(/api/v1/recommendations)과 연결됩니다. 참고로 서버에는 두 제품을 교차 비교하는 product_compare 모듈이 있지만, 지금 피그마 화면 흐름에는 대응 화면이 없습니다. 필요해지면 나중에 추가합니다.


데이터 모델

데이터 모델은 서버의 RAG 데이터 설계를 반영했습니다. 핵심 규칙은 두 가지입니다. 결측된 값은 빈 문자열이 아니라 null로 두고, 안전성은 함부로 단정하지 않습니다.

성분 모델(Ingredient)은 서버의 ingredients, restrictions, synonyms를 조인한 형태입니다. 특히 안전성 정보(safety_note)가 null이면 화면에 "안전성 확인 불가"로 표시하고, 절대 "안전함"으로 단정하지 않습니다. 화장품 성분 앱에서는 이게 신뢰성 문제라 UI 문구까지 영향을 줍니다. 제품 모델(Product)은 서버의 product와 product_ingredients 매핑을 반영해 제품 아이디, 대·소분류를 담습니다. BSTI 결과 모델(BstiResult)은 피그마 결과 화면에 맞춰 네 축의 비율(지성·건성 등), 타입 코드, 권장·주의 성분을 담습니다.

서버 응답을 다룰 때 프론트가 지키는 규칙이 있습니다. 필드가 null이면 그 항목을 화면에서 생략하고, 서버가 501을 주면 "준비 중"으로, 401을 주면 세션 만료로 보고 로그인을 유도하며, 그 외 에러는 공통 포맷의 한국어 메시지를 사용자에게 보여줍니다.


팀 협업 규칙

백엔드 팀 규칙과 동일하게 갑니다. main에 직접 커밋하지 않고 항상 브랜치를 만들어 PR을 거쳐 리뷰 후 머지합니다. 브랜치 이름은 영어로 feat/기능-설명 형식을 쓰고, 커밋 메시지도 영어 Conventional Commits(feat, fix, docs 등)를 씁니다. 다른 사람이 담당한 기능은 상의 없이 수정하지 않습니다. PR을 올리기 전에 flutter analyze, dart format, flutter test를 통과시킵니다. 화면 작업은 피그마 화면 하나를 브랜치 하나로 잡아 진행합니다.


열린 질문

팀에 확인이 필요한 것들입니다. 첫째, byeol-lab 계정을 cosmos_client 레포의 협업자(collaborator)로 추가해 주세요. 현재 push 권한이 없어 작업물을 올리지 못한 상태입니다. 둘째, 화면별 담당 분배를 위 기능 표를 기준으로 프론트 내부에서 나누면 좋겠습니다. 셋째, 백엔드 API의 세부 확정 항목은 별도 문서(프론트 API 명세)에 정리해 두었습니다.
